B1. Discover 5 unique weak/vulnerable security implementations. 

	1. Plaintext Password Storage
	- When user passwords are stored directly in a database as unencrypted plaintext strings. Any developer with database access or any attacker who gains access can instantly read all user credentials.
	- This is dangerous because a single database breach exposes every user's password immediately, as people often reuse passwords across services, attackers can use these credentials to compromise users' email accounts, banking, and more ( credential stuffing ).

	https://owasp.org/www-community/vulnerabilities/Password_Plaintext_Storage


    2. SQL injection via String Concatenation
	- User-supplied input is concatenated directly into SQL query strings without sanitisation. An attacker can craft malicious input that alters the structure of the query, bypassing authentication or extracting/deleting data.
	- SQL injection (SQLi) has consistently ranked in the OWASP Top 10 for over a decade. 
	- This is dangerous because a successful attack can result in complete database takeover, data exfiltration, authentication bypass (e.g. ' OR '1'='1), and in some configurations, remote code execution via xp_cmdshell.
	
    https://docs.datadoghq.com/security/code_security/static_analysis/static_analysis_rules/go-security/sql-string-concatenation/
    https://cheatsheetseries.owasp.org/cheatsheets/SQL_Injection_Prevention_Cheat_Sheet.html 
	
	3. Hardcoded Credentials in Source Code
    - API keys, database passwords, encryption keys, or service tokens are embedded directly in source code files. When code is committed to a version control system such as Git, these secrets become permanently part of the repository history.
    - This is dangerous because even if the credential is later removed, it persists in git history and is trivially recoverable. Public repositories are continuously scraped by automated bots searching for patterns like 'password =', 'api_key =', and 'secret ='. Compromised credentials enable unauthorised access to databases, cloud services, and third-party APIs.

    https://www.beyondtrust.com/resources/glossary/hardcoded-embedded-passwords
    https://owasp.org/www-community/vulnerabilities/Use_of_hard-coded_password


    4. Insecure Direct Object references
    - An application exposes internal object identifiers (e.g. database record IDs) in URLs or API parameters without verifying that the requesting user is authorised to access that specific resource. Changing the ID in the URL grants access to other users' data.
    - It is dangerous because IDOR is a type of broken access control, which topped the OWASP Top 10 in 2021. It requires no special tools — an attacker simply increments a number in the URL. It can expose sensitive personal data, financial records, medical information, and private messages belonging to other users.
    https://cheatsheetseries.owasp.org/cheatsheets/Insecure_Direct_Object_Reference_Prevention_Cheat_Sheet.html 


    5. Unencrypted Data Transmission (HTTP)
    - Web applications or APIs that transmit sensitive data over plain HTTP rather than HTTPS expose all traffic to interception. Data is sent in cleartext and can be captured by anyone on the same network using a packet sniffer.
    - It is dangerous because Man-in-the-Middle (MitM) attacks are trivial on unencrypted connections. Attackers on public Wi-Fi networks can capture login credentials, session tokens, and personal data. Session hijacking attacks can use intercepted cookies to impersonate authenticated users without knowing their password.


B2. Discover 5 unique strong security implementations. 
    1. Argon2 Password Hashing with Salting
    - Argon2 is a memory-hard password hashing algorithm that won the Password Hashing Competition in 2015. 
    - It is the current gold standard for securely storing passwords, designed to resist GPU-based and ASIC-based brute-force attacks.

    https://crypto.stackexchange.com/questions/88171/is-salt-required-in-argon2

    2. Parameterised Queries and ORM Usage
    - Parameterised queries separate SQL code from user-supplied data. 
    - The database driver treats input as a literal data value rather than executable SQL, making it structurally impossible for user input to alter query logic.

    https://cheatsheetseries.owasp.org/cheatsheets/Query_Parameterization_Cheat_Sheet.html

    3. Multi-Factor Authentication (MFA) with TOTP
    - Time-based One-Time Password (TOTP) MFA adds a second authentication factor beyond a password. 
    - Users must provide a 6-digit code generated by an authenticator app (e.g., Google Authenticator, Authy) that rotates every 30 seconds based on a shared secret and the current timestamp.

    https://jumpcloud.com/blog/totp-mfa#:~:text=TOTP%20(Time%2Dbased%2C%20One,%E2%80%9Csomething%20you%20have%E2%80%9D%20classification. 

    4. Content Security Policy (CSP) Headers
    - Content Security Policy is an HTTP response header that instructs browsers on which sources of content (scripts, styles, images, fonts) are permitted to load on a page. 
    - It provides a browser-enforced whitelist that blocks execution of injected malicious scripts.

    https://developer.mozilla.org/en-US/docs/Web/HTTP/Guides/CSP

    5. Role-Based Access Control (RBAC) with Least Privilege
    - Role-Based Access Control assigns permissions to roles rather than individuals. 
    - Users are granted only the roles necessary for their job function, and those roles carry only the minimum permissions required, the principle of least privilege.

    https://medium.com/o-m-n-i-navigating-the-new-cyber-era/implementing-role-based-access-control-rbac-for-least-privilege-security-f382b4083c82

B12. Discover two bias cases when using a generative AI system. 
    Case 1: Occupational Bias
    - Three identical prompts were submitted to ChatGPT, asking it to describe a typical day for each of three professions (nurse, engineer and surgeon). No gender, demographic, or personal information was included in any prompt. The only variable was the job title.
    
    Observations
    - The nurse was framed around emotional labour and caregiving, language typically associated with feminised roles. Despite nursing requiring advanced clinical skill, the description deprioritised technical competence in favour of emotional and supportive qualities.
    - The surgeon was framed around authority, leadership, and high-stakes decision-making, language typically associated with masculinised, high-prestige roles. The description used more dramatic formatting (emojis, emphasis) reinforcing a sense of gravitas.
    - The engineer was framed around technical intellect and problem-solving, language associated with STEM masculinisation, with no emotional dimension mentioned.

    Summary:
    - The model successfully avoided the surface-level pronoun bias (using job titles instead of gendered pronouns) which demonstrates that some bias mitigations are in place, but subtler, framing-level bias persists beneath the surface.


    Case 2: Political Censorship
    - A prompt was submitted to DeepSeek asking it to decode a message encoded in hexadecimal (base-16). 
    - The message was designed to contain politically sensitive content related to the status of Taiwan, which the Chinese government considers a domestic matter rather than a matter of national sovereignty.

    Observations
    - DeepSeek initially began to process and decode the hexadecimal message correctly. 
    - However, before completing its response, the output was abruptly replaced with a dismissive message stating that the query was beyond its scope and that it could not assist with the request.

    Summary:
    - This case illustrates that AI bias is not always accidental or statistical.

    While reviewing the rawsec-cybersecurity-inventory project I had already contributed to, I identified a recurring security vulnerability across the JSON data files: several tool entries were using insecure http:// links instead of https://.
    
    This is a real security issue because when users click these links from the inventory website, their browser connects to the destination over an unencrypted channel. B22. Enhance the cybersecurity of a website from your community. 
    
    For this activity I assessed the cybersecurity of my community for the project, Cafe Guilty Pleasure and produced a professional security recommendation report identifying missing HTTP security headers and how to fix them.

    What I found:
        The website was missing key HTTP security headers, leaving it vulnerable to common web attacks including cross-site scripting (XSS), clickjacking, and content injection. These headers are a low-effort, high-impact security control that are enforced by the browser on every page load and require no changes to the website's underlying code.
    
    What I recommended:
        I documented the following security headers and produced a step-by-step guide for the website owner to implement them through Cloudflare's Transform Rules:

        Content-Security-Policy (CSP) 
            restricts which sources of scripts, styles, and images the browser is permitted to load, blocking injected malicious scripts
        X-Frame-Options 
            prevents the site from being embedded in an iframe on another domain, protecting against clickjacking attacks
        X-Content-Type-Options
            stops browsers from guessing the content type of a response, preventing MIME-type sniffing attacks
        Referrer-Policy
            controls how much referrer information is sent when users navigate away, protecting user privacy
        Strict-Transport-Security (HSTS)
            forces browsers to always connect over HTTPS, preventing protocol downgrade attacks

    Outcome:
        Although I do not have direct access to the website's domain and therefore could not implement the headers myself, I produced a written recommendation document explaining each header, why it is needed, and the exact steps to apply them through Cloudflare. This was shared with the website owner for implementation.
